Критичні — пояснюють «все працює дуже погано»
1. Результати валідації миготять: той самий HTML по черзі дає N помилок і 0.
Довів прогоном: 4 послідовні validate() одного HTML → 3 issues, 0, 3, 0. Причина: RegexCache.get() повертає спільний кешований RegExp із дефолтним флагом g, а правила викликають на ньому .test() — lastIndex зберігається між викликами, і кожен другий виклик стартує з середини рядка. Вражені чеки: duplicate-styles, empty-elements-cleanup, malformed-attributes, email-unsafe-attributes, table-nesting-cleanup. Саме тому панель показує різні числа при кожному перерендері.

2. Autofix ламає посилання з query-параметрами.
email-safe-tags містить правило «квотити неквотовані значення» =\s*([^"'\s>]+) → ="$1", яке спрацьовує на будь-якому = — включно всередині вже заквотованих URL:


Вхід:  <a href="https://shop.com/sale?utm_source=email&id=42" ...>
Вихід: <a href="https://shop.com/sale?utm_source="email&id=42"" ...>
Там же <\s+ → < і \s+> → > калічать звичайний текст листа ("a < b" → "a <b").

3. Autofix стирає alt-тексти користувача на всіх картинках.
image-alt-attributes видаляє наявний alt і генерує новий з імені файлу: alt="Summer Sale — 50% off everything" → alt="Hero Banner". Кожен запуск "Auto-fix All" знищує ручну роботу.

4. paragraph-tags калічить <pre> і <picture>.
Патерн <p([^>]*)> матчить будь-який тег на "p": <pre>code</pre> → <spanre>code</pre>. Розбалансований HTML — і перевірка isValidHTMLStructure це пропускає.

5. Кнопка "Fix All" у групах severity часто не робить нічого.
getRulesBySeverity відбирає правила за задекларованою severity правила, а не за severity показаних issues. Приклад з прогону: лінк без target дає 2 warnings від email-safe-links (severity правила — error) → "Fix All (2)" на Warnings повертає «нічого не виправлено».

6. Текст із << або >> мовчки блокує всі автофікси.
isValidHTMLStructure вважає такий HTML «зламаним» і відкочує кожен фікс — <p>use operator << to shift</p>: 1 помилка, autofix → 0 змін, без жодного повідомлення.

Серйозні
display: block (з пробілом) ніколи не задовольняє чек includes("display:block") → фікс додає дубль: style="display: block; display:block".
block-element-tags переносить атрибути div на table: <div style="display:flex" class="row"> → <table ... style="display:flex" class="row"> — flex на таблиці + class, який інше правило потім видалить.
Strict Mode і половина конфіга — фікція: strictMode, allowModernCSS, maxTableNesting, requireAltText ніде не читаються. Тумблер у Settings-діалозі — no-op.
table-attributes — звалище: перевіряє і таблиці, і картинки (у suggestion для height підставлено DEFAULT_IMAGE_WIDTH); силоміць ставить width="600" height="400" на всі img без розмірів і переписує valign="middle" → "top", змінюючи верстку користувача.
Витік таймерів: конструктор EmailHTMLValidator запускає setInterval, а панель ніколи не викликає dispose() — кожен маунт вкладки лишає вічний інтервал.
Парсер губить вузли: зайвий закриваючий тег обриває parseChildren — усі наступні сиблінги зникають з AST (AST-правила їх не бачать); updatePosition() рахує line/column неправильно → криві номери рядків у репортах.
email-unsafe-attributes видаляє aria-* (суперечить власній accessibility-категорії) та id (ламає якірні посилання); патерн чека data-\w+ не матчить data-foo-bar, а патерн фікса матчить — чек і фікс розсинхронізовані.
Дрібніше: скоринг грубий (5 помилок = 0/100), результати image-alt-attributes без category/autoFixAvailable (не потрапляють у картки категорій і не мають кнопки wand), fallback невалідного регекса (?:) у replace вставив би заміну між кожним символом.
Моя рекомендація
Корінь проблеми — модуль побудований на регекс-трансформаціях цілого HTML-рядка з саморобним парсером і кешем, який робить регекси stateful. Латати по одному можна, але надійний шлях — переписати шар check/autofix на DOMParser (браузерне API, вже доступне; DOMPurify теж є в проєкті): парсити раз, перевіряти й правити атрибути на вузлах, серіалізувати назад. Це прибирає баги 1, 2, 4, 6, 7, 12 класом, а не поштучно.

Пропоную так: спершу швидкі точкові фікси критичних 1–6 (flaky-чеки, вбивчі регекси, wipe alt-ів), потім окремим кроком — міграція на DOM-підхід. Скажи, чи робити, і в якому обсязі.
