# FlexiBuilder Pro — гайд

Тут — як запустити проєкт і що робити в кожній вкладці.

---

## 1. Що це за застосунок

**FlexiBuilder Pro** — інструмент для збірки email-шаблонів: конвертація HTML/MJML, обробка зображень, відправка тестових листів, бібліотеки блоків і шаблонів, завантаження файлів на storage-провайдери (default / alphaone / ttt).

**Демо (тільки інтерфейс, без бекенду):** https://misha-vynnyk.github.io/email-helper/
Повний функціонал (сервер, робота з файлами на диску, automation-завантаження) — тільки локально або в десктоп-збірці.

---

## 2. Швидкий старт

**Потрібно:** Node.js 18+, npm. Опційно: Ollama (AI-аналіз зображень), браузер Brave (завантаження на storage).

```bash
git clone <repo>
cd email-helper
npm install     # ставить залежності root + server/ + automation/, налаштовує шляхи під ОС
npm run dev     # http://localhost:5173
```

Зупинити — `Ctrl+C`.

`npm run dev` піднімає 2 сервіси: фронтенд (5173) і бекенд (3001) — окремого AI-процесу немає, AI-проксі до Ollama вбудований у той самий бекенд. Без бекенду **не працюватимуть**: Templates, Blocks (файлова частина), відправка листів, upload на storage, серверний режим конвертера зображень і AI-аналіз (ALT/назви).

**AI-аналіз зображень** (ALT-тексти, розумні назви) не потребує окремого `npm`-скрипта — достатньо запущеної локально Ollama з мультимодальною моделлю:

```bash
ollama pull gemma3:4b
```

Хост, модель, температуру і сам промпт можна змінити прямо в UI: Converter → Налаштування → Ollama AI.

**Windows:** `npm install` налаштовує все автоматично. Якщо Brave не знайдено:
```batch
set BRAVE_EXECUTABLE_PATH=C:\Program Files\BraveSoftware\Brave-Browser\Application\brave.exe
```
Деталі: `AUTOMATION_SETUP_WINDOWS.md`.

**Кілька інстансів одночасно:**
```bash
npm run dev 0   # frontend=5173, backend=3001
npm run dev 1   # frontend=5183, backend=3011
```

Навігація в застосунку — без роутера, це вкладки згори: **Email · Blocks · Templates · Images · Converter**.

## 3. Вкладка Email — відправка тестових листів

Ліворуч — редактор листа, праворуч — налаштування SMTP-креденшлів. Індикатор угорі показує статус бекенду: **Server Online / Offline / Checking...**

### Крок 1 — обрати спосіб зберігання креденшлів
Три пігулки-перемикачі:
- **LocalStorage** — зберігається в браузері автоматично (постійно, в plaintext).
- **.env File** — підтягується з `VITE_EMAIL_USER`, `VITE_DESTINATION_EMAIL_USER`, `VITE_EMAIL_PASS` (тільки читання, поля не редагуються).
- **Manual Entry** — тільки на сесію, зникає при перезавантаженні сторінки.

⚠️ Перемикання режиму **миттєво перезаписує** усі три поля — легко випадково стерти введене.

### Крок 2 — заповнити креденшли
- **Gmail Sender** — з якого акаунта надсилати.
- **Recipient Email** — куди прийде тестовий лист.
- **Gmail App Password** — 16-символьний **App Password**, НЕ звичайний пароль. Генерується: Google Account → Security → 2-Step Verification → App passwords.

Сервер сам визначає SMTP по домену адреси відправника: Gmail → `smtp.gmail.com:587`, Outlook/Hotmail → `smtp.office365.com:587`, Yahoo → `smtp.mail.yahoo.com:465`. Для Outlook/Yahoo теж потрібен саме app-specific пароль, хоча в UI підказка згадує лише Gmail.

### Крок 3 — написати лист і надіслати
- **Email Subject** — тема листа.
- **HTML Content** — CodeMirror-редактор HTML.
- Кнопка **Send Email** активна, тільки коли: тема+HTML не порожні, обидві адреси валідні, пароль заповнений, сервер онлайн.
- **Clear Editor** очищає тільки HTML, тему залишає.

Помилки з'являються тостами: «Cannot connect to server...» (бекенд не запущено), «Email credentials are missing or invalid.» тощо.

## 4. Вкладка Blocks — бібліотека блоків

> ⚠️ **Наразі прихована в v1.0.** У `src/config/featureFlags.ts` стоїть `BLOCK_LIBRARY_ENABLED = false`. Щоб показати вкладку — змінити на `true` і зібрати заново. Це свідоме рішення релізу, а не баг.

Коли увімкнена, логіка така:

### Джерела блоків (3, зливаються в один список)
| Джерело | Що це | Залежність |
|---|---|---|
| `src/blocks/*.ts` | вбудовані блоки, зашиті в білд | немає |
| файли на диску (`data/blocks/files/`) | кастомні блоки як `.ts`-файли в налаштованих папках | потребує бекенд |
| `localStorage` | кастомні блоки в браузері | лише цей браузер, зникає при чистці кешу |

### Робота з вкладкою
- **Search** — пошук по назві/ключових словах/категорії.
- Фільтри: **джерело** (All / src/blocks / data/blocks/files / localStorage) і **категорія**.
- **Add Block** — форма: назва (до 100 симв.), категорія (Structure / Content / Buttons / Footer / Headers / Social / Custom), чекбокс «Save as TypeScript file (.ts)» (файл на диску) чи в localStorage, місце збереження, ключові слова, HTML-код (до 50 000 симв., бажано table-based з inline-стилями).
- **Storage** — керування папками: назва + абсолютний шлях (має починатися з `/`), опис; бекенд перевіряє, що шлях у межах папки Documents.
- Видалення файлового блока — **незворотне**, видаляє `.ts`-файл з проєкту.

Якщо не налаштовано жодної storage-локації — банер «No visible storage locations configured!» і файлові блоки просто не з'являться.

## 5. Вкладка Templates — бібліотека шаблонів

Шаблони підтягуються з папок на диску, які вказує користувач — вкладка **обов'язково потребує бекенд** (`npm run dev`) або встановлена програма. Без нього — помилка: «Бібліотека шаблонів потребує backend. Запустіть: npm run dev».

### Крок 1 — додати локацію
**Storage** (шестерня) → **Add New Location** → заповнити:
- **Location Name**
- **Directory Path** (+ кнопка **Browse** — тільки в десктоп/Electron-версії)
- **Description** (опційно)

Шлях має бути **абсолютним**:
- Mac/Linux: `/Users/логін/Documents/templates`
- Windows: `C:\Users\логін\templates` **без** ведучого слеша перед літерою диска

Можна додати скільки завгодно локацій; одна автоматично стає **Default** (⭐). Кожну можна сховати (👁) чи видалити (🗑) — приховані виключаються і зі списку, і з синхронізації.

### Крок 2 — синхронізувати
- **Sync New** — сканує всі активні локації і додає нові файли.
- **Refresh** — просто перезавантажує вже проіндексоване, без пересканування диска.
- Іконка sync на картці шаблону — синхронізує лише цей файл.

### Пошук і фільтри
Пошук по назві/категорії/тегах/опису/шляху. Додатково: сортування (Newest/Oldest/Name A-Z/Z-A/Category), фільтр по категорії (Newsletter/Transactional/Marketing/Internal/Other), фільтр по блоку (можна ввести `<!НазваБлоку` прямо в пошук), фільтр-чіпси по папках.

### Дії на картці шаблону
**Load** (в редактор), **Send**, копіювання, перегляд коду, **Sync**, **Edit** (метадані: назва/категорія/теги/опис — сам файл на диску не змінюється), **Delete** (видаляє тільки з індексу бібліотеки, **файл на диску залишається**).

## 6. Вкладка Images — конвертер зображень

Працює **повністю в браузері** (jsquash/WASM), без бекенду — за замовчуванням. Опційно можна перемкнути на серверну обробку.

### Завантаження
Drag-and-drop у зону «Drop images here — JPEG, PNG, WebP, AVIF & GIF supported» або кнопка **Choose Files**/**Add more**. Вставки з буфера (Ctrl+V) немає.

### Налаштування (бокова панель)
- **Format:** Original, JPG, WebP (рекомендовано), AVIF (найкраще стиснення, обмежена підтримка клієнтами), PNG (без втрат), GIF.
- **Compression mode:** Balanced (за замовч.), Premium (якість 92), Compact (якість 75), Lossless (100, тільки PNG/WebP).
- **Quality:** Auto або ручний повзунок 1–100 (мінімум 40, розблокувати нижче можна окремою кнопкою «Unlock extreme compression»).
- **Resize:** Original / пресети 1000·600·464·364px / Custom W×H («Keep ratio», «Upscale»).
- **EXIF Data** — вимкнено за замовчуванням (приватність, GPS-дані стираються).
- **Processing:** Client (в браузері) / Server (через бекенд) — **GIF завжди йде на сервер**, незалежно від перемикача, щоб зберегти анімацію.

### Пакетна обробка
До 3 файлів конвертуються одночасно (черга), автоматичні повтори при помилці (до 3 разів). Ліміт розміру: 10MB у клієнтському режимі, 50MB у серверному. Немає ZIP-експорту — «Download All» просто послідовно качає кожен файл окремо.

AI/OCR-функцій (ALT-текст, розумні назви) у цій вкладці **немає** — вони є тільки в Converter.

## 7. Вкладка Converter — HTML/MJML конвертер (головна фіча)

Найскладніша вкладка. Має два режими — перемикач **Simple / Advanced** (пігулка з бейджем «Beta» на Advanced) поруч із File Naming Bar.

### 7.1 Simple converter
1. Вставити HTML у поле «Text Editor» (Ctrl+V).
2. File Naming Bar: **File Name** (з +/– для номера), перемикач **Approve needed**, пігулки **Storage Profile — Default / AlfaOne / TTT**, тип експорту **Both / HTML only / MJML only**.
3. **Do Everything** — одна кнопка: конвертує, підставляє завантажені URL, качає файл. Або окремо в панелі виводу (**HTML Output / MJML Output / Preview**) — Export / Copy / Download.
4. **Clear All** (кошик) — скидає все.

### 7.2 Advanced converter
Розрахований на **сирий вставлений текст з Google Docs** — реконструює email-таблиці з ідіом GDocs (кольорові 1×1-клітинки → кнопки/банери, клітинки лише з бордером → розділювачі, багатоколонкові таблиці → stats-grid/record-row).

Важливо:
- Конвертується **саме оригінальний вставлений HTML**, а не те, що видно/відредаговано у видимому редакторі — правки у видимому полі на Advanced-вивід не впливають, якщо вони не зачепили сам вставлений фрагмент.
- **MJML не підтримується** в Advanced (тихо логується, не помилка).
- Пігулка **Storage Profile (Default/AlfaOne/TTT)** тут означає **набір візуальних токенів** (шрифти, паддінги, колір кнопок) для конкретного клієнта — окремих версій розмітки немає, тільки токени.
- Некритичні проблеми конвертації показуються у логах як ⚠️, напр.: «Зображення без src пропущено», «Вкладену таблицю сплющено до тексту», «Сусідні таблиці об'єднано в одну».
- Символ `§` — «нульовий» перенос рядка, спільна конвенція для обох режимів (є шпаргалка в панелі), але в Advanced він не обходить зображення.

### 7.3 Зображення в конвертері
Якщо в вставленому HTML знайдено зображення — картка «Images Detected»:
- Повзунки **Якість** (60–100%) і **Макс. ширина** (300–1200px).
- Формат на кожне зображення: **Auto / JPG / PNG / GIF** (Auto: прозорість → PNG).
- Кнопки: **Обробити все (N)**, **Upload to Storage (N)**, **Замінити (N)** / «✓ Замінено», **Очистити**.

### 7.4 Storage Upload — завантаження на провайдера
Діалог **Upload to Storage**:
- **Category** (Finance/Health) — з'являється **тільки для Default**-провайдера; AlfaOne і TTT цю категорію ховають і завжди використовують «finance» внутрішньо.
- **Folder Name** — тільки літери+цифри (напр. `ABCD123`), авто-підставляється з імені файлу.
- Список файлів: drag-переставлення, перейменування, ALT-теги.
- **Upload** → **Retry Failed** при помилці окремих файлів → **Done**.

Технічно: фронт віддає файл на `/api/storage-upload/prepare`, бекенд запускає `automation/run-upload.js` (Playwright + Brave через CDP, порт 9222/9223) → отримує публічний URL.

**Замінити (N)** — підставляє щойно завантажені URL у вихідний HTML/MJML (і ALT-атрибути). Заблоковано, якщо Output ще не експортовано або якщо upload-сесія «застаріла» (був новий Extract/Upload після останнього завантаження) — тоді просто мовчки не спрацює зі снекбар-попередженням.

### 7.5 Upload History
Панель під основним layout — до 50 останніх сесій (localStorage), розгортаються з іменами файлів, шляхами, ALT-текстом, кнопками копіювання. «Clear All» стирає все.

### 7.6 Diagnostics
Дві вкладки: **Operations Log** (що і як конвертувалось/які warnings) і **Raw HTML** (сирий вставлений HTML, з копіюванням).

### 7.7 AI-аналіз зображень (ALT-текст, розумні назви)
Кнопка **Analyze (OCR)** / **Analyze (AI)** на кожному файлі + **Analyze All (N)** у діалозі завантаження. Індикатор AI в шапці: 🟢 online / 🟠 «Ollama не знайдена» / 🔴 offline. Потребує локально запущеної Ollama з моделлю:
```bash
ollama pull gemma3:4b
```

Технічно це один запит `POST /ai-api/api/analyze` (Node-проксі `server/routes/aiProxy.js`, вбудований у основний бекенд — окремого AI-сервісу немає) до Ollama `/api/generate` з мультимодальним промптом, який просить одразу JSON `{filename, alt_text, cta}` — тобто ALT, назва файлу і CTA генеруються однією моделлю за один виклик. Відповідь кешується в пам'яті бекенду по хешу зображення.

Налаштувати можна прямо в UI (Converter → Налаштування → **Ollama AI**): хост Ollama, вибір моделі зі списку встановлених, кнопка **«Тест»**, параметри генерації (Temperature, Max tokens, Context) і повний текст системного промпту — з кнопкою «Скинути до стандартного». Якщо обрана модель не встановлена в Ollama — бекенд сам перемкнеться на першу доступну і покаже попередження.

`AltTagInput` на кожному файлі — звичайне текстове поле: ALT можна ввести вручну незалежно від AI, а AI-пропозиції (чіпси «ALT suggestions» / «CTA text») лише додають текст у те саме поле. Сам `alt=` в HTML/MJML підставляється не одразу, а тільки при завантаженні на storage або натисканні «Замінити (N)».

## 8. Часті проблеми

| Симптом | Причина |
|---|---|
| «Server Offline» / нічого не працює в Templates, Blocks, Email, upload | не запущено `npm run dev` (бекенд на 3001) |
| Templates/Blocks не бачать файлів | не додано storage-локацію, або шлях невалідний (не абсолютний) |
| Upload на storage не працює | Brave не запущено з `--remote-debugging-port` (9222/9223), або немає інтернету/доступу до storage-домену |
| AI-аналіз недоступний / «Ollama не знайдена» | не запущено застосунок Ollama, або в ній немає жодної встановленої моделі (`ollama pull gemma3:4b`) |
| Windows: Brave не знаходиться | вказати `BRAVE_EXECUTABLE_PATH` / `BRAVE_USER_DATA_DIR` |
| Advanced-конвертер «не бачить» мої правки в тексті | він бере оригінальний вставлений HTML, а не відредагований видимий текст |
| «Замінити» неактивне після Upload | Output ще не експортовано, або сесія upload застаріла |

## 9. Шпаргалка команд

```bash
npm run dev              # усе разом: frontend + backend
npm run dev-host          # те саме, доступно з мережі
npm run dev-frontend      # тільки фронтенд (5173)
npm run dev-backend       # тільки бекенд (3001)
npm run build             # прод-збірка → dist/
npm run deploy            # білд + публікація на GitHub Pages
npm run test              # Jest-тести
npm run dist:mac/dist:win # десктоп-збірка (Electron)
npm run automation:upload -- ./image.png finance --provider default
```
