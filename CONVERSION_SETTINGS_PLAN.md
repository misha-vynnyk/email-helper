# План: Налаштування конвертації + профілі клієнтів через UI

Дата: 2026-07-11 · Гілка-основа: `refactor/advanced-converter-cleanup`
Статус: **чернетка на затвердження** — жоден етап не починається без окремого "go" по ньому.

---

## 1. Мета

1. Окреме підменю (вкладка) **"Конвертація"** в налаштуваннях конвертера — усі значення, що впливають на вихідний HTML, доступні з UI.
2. Налаштування **різні для кожного профілю** (default / TTT / AlfaOne / кастомні).
3. **Створення нових профілів** під клієнта через UI: дублювання, перейменування, видалення, експорт/імпорт.
   Увесь стан налаштувань — один версіонований документ (`ConversionSettingsFile`, §3.3a), який можна вивантажити/завантажити через UI цілком: бекап, перенесення на іншу машину, шеринг колезі.
4. Фінальна фаза: ці ж профілі живлять **простий конвертер** (зараз — три форки formatter-ів з хардкодами).
5. Кожен крок — маленький, ізольований, покритий тестами, здається на ревʼю окремо. **Максимальний контроль: один етап = один комміт = одне "go".**

### Незмінні принципи (успадковані від advanced-конвертера)

- Токени — єдине джерело візуальних значень; профіль = оверрайд токенів, **ніколи** не форк розмітки.
- Налаштування зберігають **тільки явно змінені** значення (partial override). Порожнє поле = "успадковано від профілю/бази". Інакше зміна базового профілю перестане впливати на користувачів.
- Значення з документа (кольори, вирівнювання, рамки, margin-сигнали) лишаються з документа — налаштування керують лише house-стилем.

## 3. Модель даних

### 3.1 Профіль конвертації

```ts
// src/htmlConverter/profiles/types.ts (нове)
export type BuiltinBase = "default" | "ttt" | "alphaone";

export interface ConversionProfile {
  id: string;              // "default" | "ttt" | "alphaone" | uuid для кастомних
  name: string;            // відображуване імʼя ("AlfaOne", "Client X — Newsletter")
  base: BuiltinBase;       // від чого успадковується кастомний профіль
  builtIn: boolean;        // вбудовані не видаляються/не перейменовуються
  overrides: TokensOverride; // ТІЛЬКИ явно змінені користувачем значення
  storageProvider?: "default" | "ttt" | "alphaone"; // який upload-провайдер (етап 7)
  updatedAt: number;
}
```

### 3.2 Порядок розвʼязання токенів (resolution)

```
tokens (база коду)
  → builtinOverride(base)        // profiles/ttt.ts | alphaone.ts | {} для default
    → profile.overrides          // користувацькі зміни з UI
      → фінальні Tokens для buildTemplates / convertAdvancedDetailed
```

Реалізація: `resolveProfileTokens(profile): Tokens` = `mergeTokens(mergeTokens(tokens, builtin), profile.overrides)`.
⚠️ `mergeTokens` — плоский merge по секціях; для overrides з UI цього достатньо (пишемо лише листові значення), але хелпер запису має класти значення точково: `setOverride(profile, "layout.gapMarginThresholdPt", 12)` і **видаляти ключ** при скиданні (щоб успадкування відновлювалось).

⚠️ **Сигнатура `convertAdvancedDetailed`** приймає `TokensOverride`, не `Tokens` (`advanced/index.ts:31`). Повний resolved `Tokens` типово сумісний, але тоді fast-path `hasOverride` завжди true. Рішення на етапі 1: додати перевантаження/параметр, що приймає готовий `Tokens` (пропускає внутрішній `mergeTokens`/`buildTemplates`-вибір), АБО передавати злитий override — зафіксувати в diff-і етапу, вихід у будь-якому разі байт-у-байт (`templates = buildTemplates(defaultTokens)`).

### 3.3 Зберігання і міграція

- Новий ключ localStorage: `STORAGE_KEYS.CONVERSION_PROFILES` → `ConversionSettingsFile` (§3.3a): `{ version: 1, globals, profiles }` (без `activeProfileId` до етапу 7 — див. нижче).
- **Єдине джерело правди для активного профілю (етапи 1–6): `storageProfile`.** `useHtmlConverterLogic` і далі пише в `STORAGE_KEYS.STORAGE_PROFILE` при кожному перемиканні у FileNamingBar; якщо `activeProfileId` стане другим збереженим значенням, вони розсинхронізуються (міграція одноразова, а FileNamingBar живе своїм життям). Тому до етапу 7 `activeProfile` — **похідна**: `profiles.find(p => p.id === storageProfile)`. Власний `activeProfileId` (і міграція зі `STORAGE_PROFILE`) зʼявляється лише на етапі 7, коли ролі профіль/провайдер розділяються і старий ключ виводиться з ужитку.
- `oneBrSymbol` **не** переїжджає в профіль — це глобальна конвенція введення (маркер у тексті), спільна для всіх профілів. Переїздить у `globals` стору `CONVERSION_PROFILES` (§3.3a) з одноразовою міграцією значення з `UI_SETTINGS` на етапі 2; в UI — на вкладку "Конвертація". Це заразом закриває підводний камінь: `rememberUiLayout: false` **видаляє весь `UI_SETTINGS`** (`useHtmlConverterSettings.ts`, persist-ефект) — якби `oneBrSymbol` лишився там, конвертаційне налаштування мовчки скидалось би тумблером "памʼятати інтерфейс". Тест на цей сценарій — в етапі 2.
### 3.3a Файл налаштувань (`ConversionSettingsFile`)

Весь стан налаштувань конвертації — один версіонований JSON-документ, який можна вивантажити й завантажити через UI:

```ts
interface ConversionSettingsFile {
  version: 1;
  globals: { oneBrSymbol: string };   // конвенції, спільні для всіх профілів
  profiles: ConversionProfile[];      // кожен — ТІЛЬКИ partial overrides
}
```

- **Той самий обʼєкт, що лежить у `STORAGE_KEYS.CONVERSION_PROFILES`**: localStorage — робоча копія (у браузері писати на диск не можна), файл — формат бекапу/перенесення/шерингу. Ніякого другого стору.
- **Файл ніколи не містить повний знімок токенів** — тільки явні оверрайди + `version`. При завантаженні мерджиться поверх дефолтів з коду (§3.2), тому оновлення базових значень у нових версіях програми продовжують діяти для всіх користувачів. Сідінг при першому запуску створює порожні оверрайди (`overrides: {}`), а не копію дефолтів.
- **Імпорт**: zod-парсинг (схеми §3.4) з клампом і зрозумілою помилкою ("поле X поза межами, файл версії N"); міграція старших `version` при потребі. Профілі з файлу завжди додаються **копіями з новими id** — імпорт ніколи не перезаписує наявне мовчки. Розпізнаємо по схемі і файл одного профілю, і повний бандл.
- **Експорт**: download JSON (у Electron — через наявний `saveToPath`-потік). Два варіанти в UI: "Експорт профілю" та "Експорт усіх налаштувань".

На вкладці "Конвертація" всі per-profile групи (G2–G7, `placeholderHref`) позначаються бейджем "Advanced" і дизейбляться, коли активний простий режим — інакше користувач крутить поля без жодного ефекту. `oneBrSymbol` — єдине поле, що діє в обох режимах. Після етапу 8 бейджі/дизейбл знімаються.

## 5. UX / UI

### 5.1 Дворівнева структура

**Рівень 1 — вкладка "Конвертація" в наявному SettingsPopover** (швидкий доступ, 90% сценаріїв):

```

│ [Інтерфейс] [Зображення] [Конвертація]      │
├─────────────────────────────────────────────┤
│ Профіль   [ AlfaOne ▾ ]      [⚙ Керувати]   │  ← селект = activeProfile;
```

### 5.2 Ключові UX-рішення

1. **Видимість успадкування** — головне, що відрізняє це від "просто форми". Кожне поле має три стани:
   - сірий текст-значення = успадковано від бази/вбудованого профілю (плейсхолдер в інпуті, не value);
   - заповнений інпут + маркер `●` = перевизначено в цьому профілі;
   - кнопка `↺` — прибрати оверрайд (видаляє ключ, значення знову "тече" з бази).
2. **Одиниці завжди видимі** — суфікс `px`/`pt` усередині інпута (як `warningFileSizeKB` зараз). Поріг — єдине pt-поле, підказка "(і)" пояснює: *сума margin-ів двох сусідніх абзаців у GDocs*.
3. **Вбудовані профілі редаговані, але не видаляються** (🔒 = не можна rename/delete; оверрайди — можна, бо вони зберігаються окремо і скидаються).
4. **Жодних "Зберегти"** — зміни пишуться одразу (як решта налаштувань конвертера), скидання поле-за-полем + "Скинути групу" в заголовку акордеона.
5. **Селект профілю на вкладці = той самий activeProfile**, що й перемикач у FileNamingBar (після етапу 7 перемикач профілю і провайдера розділяться: у FileNamingBar лишається вибір профілю, провайдер — властивість профілю).
6. Стиль: наявна система (Tailwind + shadcn-змінні + Radix Accordion/Dialog/Select). Ніяких нових бібліотек.
7. **Stretch (не в плані, окреме рішення)**: живий міні-превʼю — фрагмент листа (абзац + бокс + сітка), що ререндериться при зміні токенів. Механіка вже є (`buildTemplates(tok)`), але це +обсяг; вирішуємо після етапу 5.

## 6. Етапи виконання

> Кожен етап: окремий комміт, зелений `tsc` + повний jest, короткий звіт із diff-оглядом. Наступний етап — тільки після вашого "go". Жодних снапшот-змін без построкової ревізії.

### Етап 1 — Інфраструктура профілів (без UI, поведінка не змінюється)

- `src/htmlConverter/profiles/`: `types.ts`, `store.ts` (load/save/seed localStorage — **без** власного `activeProfileId`, див. §3.3), `resolve.ts` (`resolveProfileTokens`).
- `useHtmlExport`: активний профіль — похідна від наявного пропа `storageProfile` (`profiles.find(p => p.id === storageProfile)`); конвертація через `resolveProfileTokens(activeProfile)` замість тернарника. Рішення по сигнатурі `convertAdvancedDetailed` (§3.2) — у цьому ж комміті. Для вбудованих профілів без оверрайдів результат **байт-у-байт** той самий (перевіряється тестом проти поточних снапшотів).
- Тести: resolution-шари, сідінг, кламп-валідація, "порожні оверрайди = поточний вихід", перемикання `storageProfile` міняє resolved-токени (немає другого джерела правди).
- **Вихід етапу:** нічого не змінилось для користувача; фундамент готовий.

### Етап 2 — Вкладка "Конвертація": каркас + G1

- Третя вкладка в `SettingsPopover` (grid-cols-3), новий `ConversionSettingsTab.tsx`. Тип `settingsTab: "ui" | "image"` розширюється і в стейті батька (`HtmlConverterPanel`), не лише в пропсах попапа.
- Селект профілю (readonly-список: 3 вбудовані, віддзеркалює `storageProfile`) + акордеон з однією групою **G1**: `oneBrSymbol` (переїзд з UI-вкладки) і `placeholderHref` (перший per-profile оверрайд — обкатка механіки `●`/`↺`; бейдж "Advanced" + дизейбл у простому режимі, §4).
- `oneBrSymbol` переїздить у `globals` стору `CONVERSION_PROFILES` з одноразовою міграцією значення з `UI_SETTINGS` (§3.3) — щоб тумблер `rememberUiLayout` більше не міг його скинути.
- Тести: RTL — рендер вкладки, зміна placeholderHref пише override і міняє вихід конвертації, `↺` повертає успадковане; oneBrSymbol працює як раніше (обидва конвертери), мігрує зі старого місця і переживає `rememberUiLayout: false`.

### Етап 3 — G2 "Ритм і відступи"

- `gapMarginThresholdPt`, `blockPadY`, `spacerPx` на вкладку.
- Тести: зміна порога перемикає `<br>`/`<br><br>` на еталонному вході; blockPadY міняє padding рядків; per-profile ізоляція (зміна в TTT не чіпає default).

### Етап 4 — G3 "Розміри тексту"

- `bodyPx`, `headlinePx`, `smallPx`, `cellPx`.

### Етап 7 — Менеджер профілів

- Діалог (рівень 2): створити (від base), дублювати, перейменувати, видалити (з підтвердженням), експорт/імпорт JSON — і одного профілю, і повного бандла `ConversionSettingsFile` (§3.3a).

- Тести: CRUD, імпорт створює копію, вбудовані захищені, вибір кастомного профілю впливає на конвертацію і на upload-провайдера, міграція старого ключа; roundtrip "експорт бандла → чистий localStorage → імпорт = той самий стан", невалідний/чужий JSON відхиляється зі зрозумілою помилкою.

### Етап 9 (опційно, тільки Electron, вирішуємо після 8) — `settings.json` на диску

- Справжній файл в userData, синхронізований з localStorage (нове IPC API читання/запису). Дає ручне редагування файлу і бекап поза браузерним сховищем.
- Окреме рішення: це нова IPC-поверхня і конфлікт-резолюшн (файл змінили руками while app running). Робимо лише якщо після етапів 7–8 буде реальна потреба; браузерна версія завжди лишається на export/import.

### Етап 8 — Простий конвертер на профілях

Найризиковіший — окремий міні-план перед стартом. Напрям:
- Витягнути хардкоди трьох formatter-ів у `SimpleProfileConfig` (шрифт, blockPad, кнопка, кольори — множина значно менша за Tokens). ⚠️ Це не 3 функції, а **6**: кожен форк експортує `formatHtml*` **і** `formatMjml*` (`formatter.ts`, `ttt/formatter.ts`, `alphaone/formatter.ts`) — MJML-гілка входить у міні-план нарівні з HTML;

- Тести: снапшоти простого конвертера (HTML **і** MJML) по всіх трьох профілях ДО рефакторингу фіксуються як еталон, ПІСЛЯ — мають збігтися байт-у-байт.

## 7. Тестова стратегія (наскрізна)

| Шар | Інструмент | Що фіксуємо |
|---|---|---|
| Resolution/store | jest unit | шари merge, seed, міграція, кламп, видалення ключа при reset |
| UI | RTL (@testing-library/react) | рендер груп, стани `●`/`↺`/успадковано, персист у localStorage, ізоляція профілів |
| Конвертація | jest e2e (наявні фікстури) | оверрайд → очікувана зміна в HTML; без оверрайдів → снапшоти незмінні |
| Регресія | наявні 7 снапшотів | етап 1 і етап 8 — нульовий diff; інші етапи — diff лише від нових фіч |

Мінімальна планка: кожен етап додає тести своєї поверхні + повний наявний jest лишається зеленим (число тестів не фіксуємо — воно росте з кожним етапом).

## 8. Ризики і відкриті питання

1. **Подвійна роль `storageProfile`** — до етапу 7 кастомних профілів не існує саме тому, що вибір профілю жорстко звʼязаний з провайдером. Розвʼязка описана в 7; до того UI показує лише 3 вбудовані, а активний профіль — похідна від `storageProfile` (§3.3), щоб не було двох джерел правди в localStorage.
1a. **Токени не діють у простому режимі до етапу 8** — мітигується бейджем "Advanced" + дизейблом груп (§4); прибрати після етапу 8.
1b. **`rememberUiLayout: false` стирає весь `UI_SETTINGS`** — закривається переїздом `oneBrSymbol` у `globals` стору профілів на етапі 2 (§3.3).
2. **`mergeTokens` і вкладені обʼєкти** — секції плоскі, але `button.padding` — рядок `"3px 5px"`; в UI розкладати на два числа чи лишити рядком? Пропозиція: два number-поля, серіалізуємо в рядок при записі.
3. **Синхронізація між вкладками браузера** — localStorage без подій; приймаємо як наявний патерн (решта налаштувань так само).
4. **Живий превʼю** — свідомо stretch, окреме рішення після етапу 5.
5. **Electron** — localStorage працює; експорт JSON через звичайний download-потік (як ZIP-експорт). Справжній `settings.json` на диску — окремий опційний етап 9, не передумова жодного з етапів 1–8.
6. **Файл налаштувань як "джерело істини"** — свідомо трактується як джерело істини *намірів користувача* (partial overrides), а не повний знімок значень: повний знімок при сідінгу назавжди заморозив би дефолти коду для наявних користувачів (§3.3a).
